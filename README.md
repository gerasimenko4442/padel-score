# Padel Score

Мобільний вебзастосунок для організації падел-турнірів: гравці, автоматичний і
справедливий розподіл пар та кортів, ротація, рахунок, статистика, п'єдестал і
PDF-звіт.

## Стек

React 19 · TypeScript (strict) · Vite · Tailwind CSS v4 · DnD Kit · jsPDF ·
localStorage · Vitest

## Швидкий старт

```bash
npm install
npm run dev       # http://localhost:5173
```

Інші команди:

```bash
npm run build      # production-збірка у dist/ (з перевіркою типів)
npm run preview    # локальний перегляд production-збірки
npm run test       # unit-тести (алгоритми, статистика, валідація)
npm run lint       # oxlint
```

## Архітектура

```
src/
  types/        доменні моделі (Player, Team, Match, Round, GameState, ...)
  algorithms/   алгоритм ротації та алгоритм формування пар — без жодних
                залежностей від UI, чисті функції, покриті тестами
  utils/        статистика, валідація, localStorage-хелпери, drag&drop-своп
  services/     оркестрація алгоритмів у повний раунд (gameService),
                генерація PDF-звіту (pdfService), персистентність (storageService)
  hooks/        useGameState — єдиний reducer, що керує всім станом гри
  components/   переюзабельні UI-блоки (картки кортів, чіпи гравців, подіум...)
  pages/        по одному екрану на крок флоу (splash → ... → фінал)
  App.tsx       маршрутизація між екранами (проста state machine, без роутера)
```

### Чому саме так

- **Алгоритм формування пар ізольований у `/algorithms`** і не імпортує нічого
  з UI — його можна тестувати (і протестовано) у повній ізоляції.
- **Статистика гравця — похідні дані, а не лічильники, що інкрементуються.**
  Єдине джерело правди — історія завершених раундів (`GameState.rounds`).
  `recomputeStats()` перебудовує статистику з нуля щоразу після завершення
  раунду. Це усуває цілий клас багів розсинхронізації і робить "reroll pairs"
  безпечним: раунд, що ще не завершений, просто не встиг потрапити в історію.
- **Reducer завжди чистий і не кидає винятків.** Усі обчислення, які можуть
  провалитися (наприклад, бракує гравців для нового раунду), виконуються у
  hook-обгортках `useGameState.ts` *до* dispatch; у reducer потрапляють лише
  вже готові, гарантовано коректні дані.
- **Ротація і формування пар — окремі кроки.** Спочатку алгоритм ротації
  чесно вирішує, хто грає, а хто відпочиває цього раунду (з урахуванням
  добровільних заявок і історії відпочинку). Тільки після цього алгоритм пар
  розподіляє тих, хто грає, по командах і кортах.

## Алгоритм формування пар

`src/algorithms/pairingAlgorithm.ts` генерує ~800 випадкових конфігурацій
розподілу гравців по командах/кортах і оцінює кожну за системою балів зі
специфікації:

| Критерій                          | Бали   |
|------------------------------------|-------:|
| Ще не грали разом як партнери       |  +100  |
| Вже грали разом як партнери         |  −300  |
| Ще не грали один проти одного       |   +80  |
| Вже грали один проти одного         |  −150  |
| Збалансований за силою матч         | до +50 |
| Справедливий розподіл відпочинку    |   +30  |
| Порушення гендерного правила        | −1000  |

Обирається конфігурація з найвищим сумарним балом. Якщо через співрозмірність
статі гравців (наприклад, 3 чоловіки та 1 жінка) коректний розподіл
математично неможливий, застосунок бере найкращий знайдений варіант і показує
попередження, замість того щоб зависнути чи зламатися.

## Прийняті рішення й припущення

Технічне завдання було дуже детальним, але кілька моментів довелося
доозначити інженерним рішенням:

- **Мова інтерфейсу — українська** (текст ТЗ, назва файлу — все вказувало на
  україномовну аудиторію); назва застосунку "Padel Score" залишена як є.
  Код, коментарі та назви змінних — англійською (стандартна практика).
- **Фіксовані пари формуються автоматично (випадково) одразу після вибору
  режиму**, з подальшим ручним редагуванням через drag & drop і кнопкою
  "Перемішати" — окремого екрана для цього не додавалось, щоб не порушувати
  10-крокового флоу з ТЗ.
- **Добровільний вибір відпочинку** доступний між раундами (від 2-го раунду і
  далі); перший раунд завжди грають усі — це найпростіший спосіб покрити
  вимогу "voluntary OR automatic", не додаючи окремий екран перед стартом.
- **"Reroll pairs"** перегенеровує лише розподіл по командах, зберігаючи вже
  визначений список тих, хто відпочиває цього раунду (це рішення ротації не
  повинно змінюватись довільно при кожному переклику).
- **Без роутера.** Флоу — це послідовний wizard/game, тож навігація — це
  проста state machine в `App.tsx`, а не react-router. Це також прибирає
  типові проблеми з маршрутизацією на GitHub Pages.

## Деплой на GitHub Pages

У репозиторії вже є GitHub Actions workflow (`.github/workflows/deploy.yml`),
що збирає застосунок і публікує `dist/` на GitHub Pages при пуші в `main`.

Достатньо:

1. Запушити цей проєкт у новий репозиторій на GitHub.
2. У Settings → Pages вибрати джерело **GitHub Actions**.
3. Запушити в `main` — деплой запуститься автоматично.

`vite.config.ts` використовує відносний `base: './'`, тож збірка коректно
працює на будь-якому під-шляху (`username.github.io/repo-name/`) без
додаткового налаштування.

## Тести

```bash
npm run test
```

31 unit-тест покриває: алгоритм формування пар (відсутність дублікатів,
дотримання гендерних правил, уникнення повторів партнерів/суперників,
коректна поведінка при неможливому співвідношенні статей), алгоритм ротації
(справедливий розподіл відпочинку, пріоритет тим, хто щойно відпочивав),
перерахунок статистики та валідацію.
